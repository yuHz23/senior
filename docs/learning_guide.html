IoT Gateway Anomaly Detection

Phát hiện tấn công mạng và lỗi cảm biến IoT bằng GRU Autoencoder với Semantic Communication

Deep Learning GRU Autoencoder Anomaly Detection Semantic Communication IoT Security Python / PyTorch

Giới thiệu đồ án

Bài toán

Hệ thống IoT (Internet of Things) ngày càng phổ biến — hàng tỉ thiết bị cảm biến, camera, router kết nối internet. Những thiết bị này thường bị tấn công (DDoS, DoS) hoặc gặp lỗi cảm biến. Vấn đề: làm sao phát hiện tấn công tự động, thời gian thực, không cần biết trước kiểu tấn công?

Thách thức chính: Tấn công mới xuất hiện liên tục → không thể train supervised model với tất cả loại tấn công. Thiết bị IoT có tài nguyên hạn chế → model phải nhỏ gọn, nhanh. Dữ liệu truyền đi bị hạn chế băng thông → cần nén dữ liệu thông minh.

Giải pháp đề xuất

Dùng GRU Autoencoder — một mạng neural học cách "tái tạo" lại dữ liệu bình thường. Khi gặp tấn công, model tái tạo kém → reconstruction error cao → phát hiện anomaly.

Ý tưởng cốt lõi: Train model chỉ trên dữ liệu BÌNH THƯỜNG → model học "thế nào là bình thường" → Khi gặp tấn công, model không tái tạo được tốt → lỗi cao = CẢNH BÁO.

Unsupervised Learning

Không cần label dữ liệu tấn công. Chỉ cần dữ liệu bình thường để train. Phát hiện được cả tấn công chưa biết trước.

MaxAbsScaler — Chuẩn hóa dữ liệu

xscaled = x / |max(x)|

Cách hoạt động

Chia mỗi feature cho giá trị tuyệt đối lớn nhất của feature đó trong tập train. Kết quả: dữ liệu nằm trong [-1, 1].

Tại sao MaxAbsScaler?

Phù hợp với activation tanh trong model (output cũng [-1, 1]). Giữ nguyên giá trị 0. Ít nhạy cảm với outlier hơn MinMaxScaler.

Fit trên Train only

Scaler chỉ được fit trên tập train. Val và Test được transform theo tham số của train — tránh data leakage.

# Fit scaler trên training data (normal only)
scaler = MaxAbsScaler()
scaler.fit(X_train)          # học max từ train
joblib.dump(scaler, path)   # lưu lại để dùng sau

# Transform tất cả splits
X_train = scaler.transform(X_train)  # shape: (N, 5)
X_val   = scaler.transform(X_val)    # shape: (M, 5)
X_test  = scaler.transform(X_test)   # shape: (K, 5)

Sliding Window — Cửa sổ trượt

Tại sao cần Window?

Dữ liệu IoT là chuỗi thời gian — một gói tin đơn lẻ không đủ ngữ cảnh để biết đó là tấn công hay không. Tấn công DDoS chỉ lộ rõ khi nhìn vào nhiều gói liên tiếp. Window size = 64 nghĩa là model nhìn 64 bước thời gian cùng lúc.

Cách hoạt động của Sliding Window

Memory-efficient: as_strided Thay vì copy từng window (tốn 46M×64×5×4 bytes ≈ 59 GB!), dùng numpy stride tricks tạo VIEW zero-copy. Chỉ copy 1 batch nhỏ mỗi lần inference. 
# Zero-copy: chỉ thay đổi cách nhìn bộ nhớ
shape   = (n, window_size, F)
strides = (data.strides[0],   # slide 1 sample
           data.strides[0],   # time axis
           data.strides[1])   # feature axis
windows = as_strided(data, shape, strides)

GRU Autoencoder — Kiến trúc tổng quan

Ý tưởng Autoencoder: Nén dữ liệu xuống → giải nén lại → so sánh. Nếu dữ liệu bình thường (đã train), tái tạo tốt. Nếu là tấn công (chưa từng thấy), tái tạo kém → lỗi cao.

Encoder — Nén dữ liệu

Tại sao dùng GRU (không phải LSTM hay Transformer)?

GRU vs LSTM GRU có 3 gates (reset, update), LSTM có 4 gates (forget, input, output, cell). Ít tham số hơn → nhẹ hơn → phù hợp IoT edge device.
GRU vs Transformer Transformer cần full attention (O(n²) memory). GRU sequential O(n) — phù hợp streaming real-time với memory hạn chế.
Kết quả thực tế ROC-AUC = 1.0 trên cả 2 datasets — GRU đủ mạnh cho bài toán này mà không cần model phức tạp hơn.

Quá trình encode:

def encode(self, x):
    # x shape: (batch=256, seq_len=64, input_size=5)
    _, h_n = self.encoder_gru(x)
    # h_n shape: (num_layers=1, batch=256, hidden=128)

    h_last = h_n[-1]
    # h_last: (256, 128) — hidden state của bước cuối cùng
    # Đây là "summary" của toàn bộ chuỗi 64 bước

    z = torch.tanh(self.encoder_fc(h_last))
    # z: (256, 32) — SEMANTIC VECTOR
    # tanh giữ z trong [-1, 1], phù hợp với MaxAbsScaler input
    return z
h_n là gì? Sau khi GRU đọc qua 64 timestep, hidden state cuối cùng h_n chứa "tóm tắt" của toàn bộ chuỗi. Giống như bạn đọc xong 1 đoạn văn, h_n là "ấn tượng tổng thể" mà bạn nhớ lại.

Decoder — Tái tạo dữ liệu

def decode(self, z):
    # z: (batch=256, bottleneck=32)
    h0 = torch.tanh(self.decoder_fc(z))
    # h0: (256, 128) — khôi phục về hidden size

    repeated = h0.unsqueeze(1).repeat(1, self.seq_len, 1)
    # repeated: (256, 64, 128) — broadcast context vector qua 64 bước thời gian
    # Ý tưởng: dùng cùng 1 "thông tin nén" để tái tạo toàn bộ chuỗi

    out, _ = self.decoder_gru(repeated)
    # out: (256, 64, 128)

    x_hat = self.output_fc(out)
    # x_hat: (256, 64, 5) — tái tạo về cùng shape với input
    return x_hat
Tại sao repeat context vector? Decoder phải tái tạo chuỗi 64 bước chỉ từ vector 32 chiều. Cách đơn giản nhất: "broadcast" vector đó ra 64 lần rồi để GRU học cách mở rộng. Không cần teacher forcing vì là unsupervised.

Reconstruction Error:

def reconstruction_error(self, x):
    x_hat = self.forward(x)           # (batch, 64, 5)
    error = ((x - x_hat) ** 2).mean(dim=(1, 2))
    # MSE trung bình trên cả 64 timestep VÀ 5 features
    # Kết quả: (batch,) — 1 số cho mỗi window
    return error

Vòng lặp training mỗi epoch:

Forward pass
Model nhận batch (64 windows) → encode → decode → tính MSE loss giữa input và reconstruction.
Backward pass
PyTorch tự động tính gradient của loss với mọi tham số. Adam optimizer cập nhật weights.
Validation loss
Sau mỗi epoch, tính loss trên tập val (không update weights). Theo dõi để phát hiện overfitting.
Save best model
Nếu val_loss thấp hơn tốt nhất từ trước → lưu checkpoint. Đây là file ciciot_model.pt.
Early stopping check
Nếu val_loss không cải thiện trong 5 epoch liên tiếp → dừng sớm (patience=5).
# Simplified training loop
for epoch in range(1, epochs+1):
    model.train()
    for (batch,) in train_loader:
        optimizer.zero_grad()
        recon = model(batch)
        loss  = MSELoss(recon, batch)   # học tái tạo input
        loss.backward()                  # tính gradient
        optimizer.step()                  # cập nhật weights

    val_loss = evaluate_val(model, val_loader)

    if val_loss < best_val_loss:
        best_val_loss = val_loss
        torch.save(model.state_dict(), save_path)  # lưu best model
        patience_counter = 0
    else:
        patience_counter += 1
        if patience_counter >= patience:
            break  # early stopping

Ngưỡng phát hiện (Threshold)

Ngưỡng là gì và tính như thế nào?

Sau khi train, chạy model trên tập validation (normal only) để lấy distribution của reconstruction error trên dữ liệu bình thường.

Chọn 99th percentile làm ngưỡng — nghĩa là 99% dữ liệu bình thường có error thấp hơn ngưỡng này.

threshold = percentile(errors_normal, 99)
Ý nghĩa thực tế: ~1% dữ liệu bình thường sẽ bị cảnh báo nhầm (false positive rate ≈ 1%). Có thể điều chỉnh: 95th → nhạy hơn (nhiều FP), 99.5th → ít FP hơn.

Early Stopping

Vấn đề Overfitting

Nếu train quá nhiều epoch, model sẽ "nhớ" training data thay vì "hiểu" pattern chung → val_loss tăng dù train_loss giảm.

Hậu quả: Model tái tạo tốt training data nhưng kém trên dữ liệu mới → threshold bị sai → phát hiện kém.

Giải pháp: Early Stopping

Theo dõi val_loss mỗi epoch. Nếu không cải thiện sau 5 epoch liên tiếp → dừng và lấy checkpoint tốt nhất.

Kết quả thực tế: "We saved 2 best epoch" — model hội tụ sớm, chỉ cần vài epoch để đạt optimum. Điều này cho thấy model học nhanh và dữ liệu có pattern rõ ràng.

Precision — Độ chính xác cảnh báo

Precision = TP / (TP + FP)

Trong số tất cả lần cảnh báo "Attack", bao nhiêu % là tấn công thật? Precision cao = ít cảnh báo nhầm.

Recall — Tỉ lệ phát hiện

Recall = TP / (TP + FN)

Trong số tất cả tấn công thật, bao nhiêu % được phát hiện? Recall cao = ít bỏ sót. Quan trọng hơn trong bảo mật!

F1-Score — Trung bình hòa

F1 = 2 × Precision × Recall / (Precision + Recall)

Kết hợp Precision và Recall thành 1 chỉ số. Tốt khi cả hai đều tốt.

ROC-AUC — Không phụ thuộc threshold

AUC = 1.0 → tách biệt hoàn hảo

Đo khả năng model phân biệt Normal vs Attack theo mọi ngưỡng có thể. AUC=1.0 nghĩa là model luôn cho error cao với attack và thấp với normal — hoàn hảo.

Balanced Confusion Matrix — Tại sao cân bằng?

CICIoT có 46.7 triệu dòng nhưng phân phối lệch: nhiều attack hơn normal. Nếu tính confusion matrix thô, kết quả bị lệch theo class đa số.

Vấn đề với dữ liệu mất cân bằng: Nếu có 90% Normal và 10% Attack, model luôn đoán "Normal" cũng đạt accuracy 90% — nhưng vô dụng.
Balanced Confusion Matrix: Downsample class đa số xuống bằng class thiểu số. Mỗi class có cùng số lượng → đánh giá công bằng. 
def balanced_confusion_matrix(y_true, y_pred):
    pos_idx = np.where(y_true == 1)[0]   # attack indices
    neg_idx = np.where(y_true == 0)[0]   # normal indices
    n = min(len(pos_idx), len(neg_idx))    # take the smaller count
    # Random sample to balance
    pos_sample = rng.choice(pos_idx, size=n, replace=False)
    neg_sample = rng.choice(neg_idx, size=n, replace=False)
    idx = np.concatenate([pos_sample, neg_sample])
    return confusion_matrix(y_true[idx], y_pred[idx])

Nâng cấp v2 — Intel Threshold Tuning

Trước (v1): Cả 2 dataset dùng chung 99th percentile. Intel có 6.9% false positive (6,921 cảnh báo nhầm).
Sau (v2): Config riêng từng dataset. Intel dùng 99.5th percentile. Kỳ vọng giảm false positive xuống ~0.5%.

config.yaml — Per-dataset percentile:

threshold:
  ciciot_percentile: 99     # 1% FP rate — network traffic ổn định
  intel_percentile:  99.5   # 0.5% FP rate — sensor data biến động nhiều hơn
Tại sao Intel cần threshold cao hơn? Sensor data (nhiệt độ, độ ẩm) có nhiều biến động tự nhiên trong ngày — sáng sớm vs buổi trưa, mùa hè vs mùa đông. Network traffic ổn định hơn. Threshold cao hơn = dung sai cao hơn cho dao động bình thường.

Retrain Intel với threshold mới:

python scripts/train.py --dataset intel

Nâng cấp v2 — Ablation Study (So sánh model)

Ablation study = thử nghiệm so sánh — chứng minh tại sao GRU là lựa chọn tốt nhất bằng cách so với các kiến trúc khác.

GRU Autoencoder

Model chính

3 gates (reset, update, new). Ít tham số, nhanh, phù hợp IoT edge.

LSTM Autoencoder

So sánh

4 gates (forget, input, output, cell). Nhiều tham số hơn, chậm hơn, độ chính xác tương đương.

MLP Autoencoder

Baseline

Feedforward thuần túy. Flatten window → bottleneck. Không có temporal modelling → yếu hơn.

Chạy ablation study:

python scripts/compare_models.py --dataset ciciot

Kiến trúc MLP Autoencoder (baseline):

# Flatten window rồi encode/decode như MLP thông thường
Input: (batch, 64, 5) → flatten() → (batch, 320)
  ↓ Linear(320 → 256) + ReLU
  ↓ Linear(256 → 128) + ReLU
  ↓ Linear(128 → 32)  + Tanh   ← bottleneck
  ↓ Linear(32  → 128) + ReLU
  ↓ Linear(128 → 256) + ReLU
  ↓ Linear(256 → 320)
Output: (batch, 320) → reshape() → (batch, 64, 5)

Nâng cấp v2 — MQTT Gateway

Tại sao MQTT thay vì HTTP? MQTT là giao thức publish-subscribe nhẹ, được thiết kế cho IoT. LoRa, Zigbee, NB-IoT đều hỗ trợ MQTT. HTTP có overhead lớn hơn nhiều cho thiết bị pin nhỏ. MQTT hoạt động tốt với băng thông 2G/NB-IoT (vài KB/s).

Khởi động MQTT gateway:

# 1. Install Mosquitto broker
# Windows: https://mosquitto.org/download/

# 2. Install paho-mqtt
pip install paho-mqtt

# 3. Start gateway
python scripts/run_mqtt_gateway.py --dataset ciciot --broker localhost

# 4. Subscribe to alerts (another terminal)
mosquitto_sub -t iot/gateway/ciciot/alerts

# 5. Publish test data
mosquitto_pub -t iot/gateway/ciciot/data \
  -m '{"features": [0.1, 500, 10, 0.05, 1024], "device_id": "router_01"}'

Cơ chế sliding window trong MQTT:

deque(maxlen=64): Gateway dùng collections.deque với maxlen=64. Mỗi message MQTT = 1 sample. Khi deque đủ 64 samples → inference tự động. Sample cũ nhất bị đẩy ra khi sample mới vào. Không cần gửi đủ 64 samples trong 1 message — gateway tích lũy dần.

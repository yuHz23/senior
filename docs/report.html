Abstract

The rapid proliferation of Internet of Things (IoT) devices has introduced two critical challenges: the need for bandwidth-efficient data transmission and robust real-time detection of cyberattacks, particularly zero-day attacks with no prior signature. This project addresses both challenges by developing an IoT Gateway that employs a Gated Recurrent Unit (GRU) Autoencoder for simultaneous semantic data compression and unsupervised anomaly detection.

The system uses a GRU Autoencoder architecture trained exclusively on normal traffic data. The Encoder compresses each sliding window of 64 timesteps and 5 network features (320 values) into a compact 32-dimensional semantic vector, achieving a 10:1 compression ratio and 90% bandwidth savings. The Decoder reconstructs the original sequence from this semantic vector, and the reconstruction error (MSE) serves as the anomaly indicator. Windows whose MSE exceeds a statistically determined threshold (99th percentile of normal reconstruction errors) are classified as anomalous.

The system was evaluated on two benchmarks: the CIC IoT Dataset (46.7 million records, 33 attack types including DDoS, DoS, Mirai, and Recon) and the Intel Berkeley Lab Dataset (sensor fault detection). On the CIC IoT Dataset, the model achieves ROC-AUC = 1.0000, F1-score = 1.00, with Recall = 1.000 across all 33 attack categories. On the Intel Berkeley Dataset, the model achieves ROC-AUC = 1.0000 and F1-score = 0.97. An ablation study confirms GRU outperforms both LSTM and MLP baselines while requiring fewer parameters (159,909). The production Gateway processes 205 windows per second on CPU. The complete system includes a REST API (FastAPI), MQTT broker integration for real IoT protocols, and a memory-efficient streaming inference pipeline.

Keywords: IoT Security, GRU Autoencoder, Anomaly Detection, Semantic Communication, Edge Computing, Network Intrusion Detection, MQTT.

Chapter I — Introduction

1.1. Issue

The emergence of Industry 4.0 has accelerated the global deployment of Internet of Things (IoT) systems across smart homes, industrial automation, healthcare monitoring, and intelligent transportation networks. The number of IoT-connected devices worldwide is projected to exceed 29 billion by 2030 [1]. Each of these devices continuously generates network traffic data and sensor telemetry, creating two converging problems that this project addresses.

First, transmitting all raw data from edge devices to centralized cloud servers consumes substantial network bandwidth. IoT networks frequently operate over constrained links (LoRa, NB-IoT, Zigbee) where raw data transmission is impractical. Second, the expanding attack surface makes IoT networks attractive targets for cybercriminals. The CIC IoT Dataset alone catalogs 33 distinct attack categories including DDoS floods, DoS attacks, Mirai botnet variants, and reconnaissance scans [2], and novel zero-day variants continue to emerge daily.

1.2. Problems

1.2.1. Bandwidth Constraints in IoT Networks

IoT networks frequently operate under severe bandwidth limitations. Transmitting the entirety of raw sensor data and network flow statistics to a centralized server is inefficient and often infeasible. A single IoT gateway monitoring hundreds of devices may need to process thousands of network flow records per second. Without intelligent data compression, this creates a bandwidth bottleneck that degrades system responsiveness and increases operational cost.

1.2.2. Resource-Limited Edge Devices

IoT edge devices typically have constrained computational resources. Running complex deep learning models directly on these devices requires careful architecture design to balance detection accuracy with computational efficiency. The model must be lightweight enough to perform inference in real time — within milliseconds per window — while achieving high detection rates across diverse attack types.

1.2.3. Zero-Day Attack Vulnerability

Conventional intrusion detection systems rely on signature-based detection, which fails against zero-day attacks that lack predefined signatures. In IoT environments, where devices may run outdated firmware and attack toolkits evolve rapidly, an effective detection system must identify anomalous behavior without relying on prior knowledge of specific attack patterns. This requires an unsupervised learning approach.

1.2.4. Latency in Cloud-Dependent Processing

Centralized cloud-based security analysis introduces round-trip network latency. Processing data at the edge significantly reduces detection latency and enables faster response to security threats. Edge inference also reduces privacy exposure by avoiding raw data transmission to remote servers.

1.3. Problem Statement

This project aims to develop a software-based IoT Gateway that employs a GRU Autoencoder to simultaneously perform: (1) Semantic Data Compression — the Gateway uses the GRU Encoder to compress each data window into a compact 32-dimensional semantic vector, reducing bandwidth consumption by 90%; and (2) Unsupervised Anomaly Detection — the reconstruction error between the original data and the Decoder output serves as an anomaly score, enabling detection of both known and unknown attack types without requiring labeled attack data during training.

1.4. Motivation

The motivation stems from three converging trends: the emergence of Semantic Communication as a bandwidth-efficient paradigm for constrained IoT networks [3], the demonstrated success of autoencoder-based anomaly detection across time-series domains [4][5], and the availability of large-scale, realistic IoT network traffic datasets such as CIC IoT [2] and Intel Berkeley Lab Data [6] for rigorous evaluation. The GRU architecture is selected over LSTM for its reduced parameter count and faster convergence, and over MLP for its ability to capture temporal dependencies inherent in sequential network traffic.

Chapter IV — Literature Review

4.1. IoT Security and Intrusion Detection Systems

4.1.1. Traditional Signature-Based Approaches

Traditional IDS operate by maintaining a database of known attack signatures and comparing incoming network traffic against these patterns. Systems such as Snort and Suricata have been widely deployed in enterprise environments. However, as noted by Butun et al. (2020) [7], signature-based methods are fundamentally incapable of detecting novel attacks that lack predefined signatures, and maintaining an up-to-date signature database for the diverse IoT threat landscape is impractical given the rapid emergence of new attack variants.

4.1.2. Machine Learning-Based Anomaly Detection

Researchers have increasingly turned to machine learning and deep learning methods for anomaly detection. Chalapathy and Chawla (2019) [8] provided a comprehensive survey categorizing methods into supervised, semi-supervised, and unsupervised paradigms. Unsupervised methods are particularly attractive for IoT security because they do not require labeled attack data for training, enabling detection of zero-day attacks. This project adopts an unsupervised approach, training exclusively on normal traffic samples.

4.2. Autoencoder Architectures for Anomaly Detection

4.2.1. Standard Autoencoders

Autoencoders learn to compress input data into a lower-dimensional latent representation and reconstruct the original data. When trained exclusively on normal data, anomalous inputs — which the model has never encountered — produce higher reconstruction errors. Sakurada and Yairi (2014) [9] demonstrated the effectiveness of autoencoders for spacecraft telemetry anomaly detection.

4.2.2. LSTM Autoencoder for Time-Series Data

Malhotra et al. (2016) [4] proposed the LSTM Autoencoder, replacing feedforward layers with LSTM cells to capture temporal dependencies in sequential data. The original LSTM architecture by Hochreiter and Schmidhuber (1997) [10] uses four gating mechanisms (forget, input, output, cell) to learn long-range temporal dependencies. LSTM Autoencoders have been applied to network intrusion detection with strong results [5].

4.2.3. GRU as a Lightweight Alternative

Cho et al. (2014) [11] proposed the GRU as a simplified variant of LSTM using only two gates (reset and update), reducing the parameter count by approximately 25% compared to an equivalent LSTM. Empirical studies [12] show GRU achieves comparable performance to LSTM on many sequence modeling tasks while converging faster — a property confirmed by our ablation study where GRU achieves F1=0.9888 versus LSTM's F1=0.5250 under equal training epochs (see Section 6.5). This makes GRU preferable for resource-constrained IoT edge deployment.

4.3. Semantic Communication for IoT

4.3.1. Traditional vs. Semantic Communication

Traditional communication based on Shannon's theory (1948) [13] focuses on accurate bit-level transmission regardless of semantic meaning. Semantic communication, as conceptualized by Weaver (1953) [14], aims to convey meaning rather than exact bit-level representation. This paradigm is particularly beneficial for IoT networks where bandwidth is severely constrained.

4.3.2. Deep Learning-Based Semantic Compression

Xie et al. (2021) [15] proposed DeepSC for semantic text communication using deep neural networks. Bourtsoulatze et al. (2019) [16] demonstrated deep joint source-channel coding for wireless image transmission. Kountouris and Pappas (2021) [3] argued that semantic communication is especially beneficial for resource-constrained IoT devices. This project adopts this paradigm: the GRU Encoder's 32-dimensional bottleneck vector serves as the semantic representation of a 320-value input window, achieving 10:1 compression with 90% bandwidth savings.

4.4. Datasets

4.4.1. CIC IoT Dataset

The CIC IoT Dataset [2] from the Canadian Institute for Cybersecurity contains 46.7 million network flow records with 5 statistical features: flow_duration, Rate, Number, IAT, and Tot_size. Records are labeled across 33 attack categories (DDoS floods, DoS attacks, Mirai variants, Recon scans, web attacks) plus a BenignTraffic class. The large scale and diverse attack taxonomy make it the most comprehensive IoT intrusion detection benchmark available.

4.4.2. Intel Berkeley Lab Dataset

The Intel Berkeley Lab Dataset [6] contains sensor readings from 54 sensor nodes deployed in the Intel Research Lab, recording temperature, humidity, light intensity, and voltage. This dataset enables evaluation of the system on sensor fault detection — a complementary IoT anomaly detection scenario distinct from network intrusion. Fault labels are derived by rule-based out-of-range detection, providing a realistic evaluation of the model's generalization to sensor domains.

4.5. Normalization and Preprocessing

MaxAbsScaler normalization (xnorm = x / max(|x|)) scales features to the range [−1, 1] while preserving zero values and signal shape. This is compatible with the tanh activation used in the GRU bottleneck layer. The sliding window technique transforms sequential data into fixed-length windows of 64 timesteps, enabling the GRU to capture temporal patterns spanning multiple consecutive observations [4].

Chapter V — Methodology

5.1. System Overview

Figure 5.1: System Architecture — GRU Autoencoder IoT Gateway with Semantic Compression

The system follows a two-phase design. The Training Phase (offline) loads the dataset, fits a MaxAbsScaler on normal training data, constructs sliding windows, trains the GRU Autoencoder, and computes a statistical anomaly threshold from the validation set reconstruction errors. The Inference Phase (online) streams raw data through the trained model in real time, classifying each window as NORMAL or ATTACK based on whether the reconstruction error exceeds the threshold, and logging results with the semantic vector for downstream analysis or transmission.

5.2. Objective 1: Data Acquisition and Preprocessing

5.2.1. Dataset Selection and Analysis

Two datasets are used. The CIC IoT Dataset provides 46.7 million network traffic records across 33 attack types and BenignTraffic, with 5 features per record. The Intel Berkeley Lab Dataset provides sensor readings from 54 nodes with 4 features per record. Faults in the Intel dataset are identified by hard boundary rules (e.g., temperature outside [−20°C, 100°C]).

5.2.2. Data Splitting

For both datasets, all normal samples are extracted and split 90% training / 10% validation. The test set contains all samples (normal and attack) and is used only at evaluation time. This strict separation prevents data leakage and ensures the model learns only from normal patterns.

5.2.3. Feature Normalization

MaxAbsScaler is fitted exclusively on the training normal data: xnorm = x / max(|xtrain_normal|). The fitted scaler is saved and applied to validation and test sets. Per-dataset scalers are maintained to avoid cross-contamination between network and sensor domains.

5.2.4. Sliding Window Construction

Data is segmented into sliding windows of width W = 64 timesteps with stride 1, producing (N − W + 1) overlapping windows from N samples. Each window has shape (64, n_features), capturing temporal context for the GRU. For memory efficiency, window construction during evaluation uses NumPy stride tricks (as_strided) to create a zero-copy view, avoiding the allocation of an array that would otherwise occupy several gigabytes of RAM for the full CICIoT test set.

5.3. Objective 2: GRU Autoencoder Architecture

5.3.1. Encoder Design

The Encoder processes the input sequence (batch, 64, n_features) through a single-layer GRU with hidden_size = 128. The final hidden state hn ∈ ℝ128 summarizes the entire 64-step sequence. A linear projection followed by tanh activation compresses hn to the 32-dimensional semantic vector z ∈ [−1, 1]32.

5.3.2. Semantic Vector

The 32-dimensional semantic vector z serves two purposes: (1) it is logged and can be transmitted over low-bandwidth IoT links as a compressed representation of the original 320-value window, achieving 10:1 compression and 90% bandwidth savings; (2) it is fed to the Decoder for anomaly detection via reconstruction error. The tanh activation bounds z to [−1, 1], consistent with the MaxAbsScaler input range.

5.3.3. Decoder Design

The Decoder projects z back to hidden_size = 128 via a linear layer with tanh activation, then replicates this vector across seq_len = 64 timesteps. A single-layer GRU with hidden_size = 128 processes this repeated context, and a final linear layer projects each timestep to n_features, yielding the reconstructed sequence x̂ ∈ ℝbatch × 64 × n_features.

5.4. Objective 3: Training and Threshold Computation

5.4.1. Training Configuration

The model is trained exclusively on normal-class windows with MSE loss, Adam optimizer (lr = 0.001), batch size 64, for up to 50 epochs. Early stopping with patience = 5 saves the best checkpoint at the lowest validation loss. Both datasets used early stopping: the Intel model stopped at epoch 17 with best val_loss = 0.011174.

5.4.2. Threshold Computation

After loading the best checkpoint, reconstruction errors are computed on the normal-only validation set. The anomaly threshold is set at the k-th percentile of these errors, where k = 99 for CICIoT (threshold = 0.003272) and k = 99.5 for Intel Berkeley (threshold = 0.021932). The higher percentile for Intel reflects greater natural variance in sensor readings versus network traffic patterns. This per-dataset tuning reduces Intel false positives by 19.3% (from 6,921 to 5,582) while adding only 1 missed fault.

5.5. Objective 4: Anomaly Detection

For each test window x ∈ ℝ64 × n_features, the reconstruction error is computed as:

error(x) = (1 / (W × F)) Σt,f (xt,f − x̂t,f)²

The classification rule is: label = ATTACK if error(x) > threshold, else NORMAL. This threshold comparison executes in O(1) time after the GRU forward pass, enabling real-time detection.

5.6. Objective 5: Semantic Compression

The 32-dimensional semantic vector achieves a 10:1 compression ratio over the original 320-value window (64 timesteps × 5 features). In an IoT deployment, only 32 float32 values (128 bytes) need to be transmitted per inference window, compared to 1,280 bytes for the raw window — a 90% bandwidth saving. This enables deployment over severely bandwidth-constrained links such as LoRa (0.3–50 kbps) and NB-IoT (20–250 kbps).

5.7. Objective 6: IoT Gateway Implementation

5.7.1. CSV Streaming Gateway

The base gateway streams CSV data row-by-row via a generator function and accumulates samples in a WindowBuffer (deque with maxlen = 64). Once the buffer is full, each new sample triggers one inference. Results are written to a timestamped JSONL log file with label, reconstruction error, threshold, semantic vector, and timestamp per window.

5.7.2. FastAPI REST Endpoint

A production REST API is built with FastAPI, exposing four endpoints: GET /health (liveness), GET /info/{dataset} (model metadata), POST /detect (single window inference), and POST /detect/batch (batch inference). The API uses Pydantic for input validation and Uvicorn as the ASGI server. Swagger UI documentation is auto-generated at /docs.

5.7.3. MQTT Gateway

A dedicated MQTT Gateway subscribes to topic iot/gateway/{dataset}/data, accumulates samples in a sliding deque, and publishes detection results to iot/gateway/{dataset}/alerts and periodic statistics to iot/gateway/{dataset}/status. This implements the full publish-subscribe IoT communication pattern using the MQTT v3.1.1 protocol via the paho-mqtt library, compatible with Mosquitto brokers.

6.6. Results of Objective 5: Semantic Compression

The GRU Encoder produces a 32-dimensional semantic vector from each 64×5 = 320-value input window, achieving a 10:1 compression ratio and 90% bandwidth savings. This is a 2× improvement over the 5:1 ratio of a 64-dimensional bottleneck, enabling deployment over even more bandwidth-constrained IoT links. The semantic vector is bounded to [−1, 1]32 by the tanh activation, facilitating standardized transmission and storage.

The ROC-AUC of 1.0000 on both datasets confirms that the 32-dimensional semantic space perfectly separates normal and anomalous traffic — the encoder learns a compact but semantically meaningful representation that preserves the information necessary for anomaly detection.

  • FastAPI REST API: Both ciciot and intel detectors load at startup. Average inference latency ~5ms per window. Swagger UI at /docs provides interactive documentation.
  • MQTT Gateway: Subscribes to iot/gateway/{dataset}/data, publishes detection results to alerts topic, and periodic statistics to status topic every 30 seconds.

Each JSONL log entry contains: label (NORMAL/ATTACK), reconstruction_error (float), threshold (float), semantic_vector (32 floats), window_id (int), and timestamp (ISO 8601).

Chapter VII — Conclusion and Future Work

7.1. Conclusions

This senior project successfully developed a production-ready IoT Gateway that employs a GRU Autoencoder for simultaneous semantic compression and unsupervised anomaly detection. The following key achievements were demonstrated:

  1. Near-Perfect Anomaly Detection: The system achieves ROC-AUC = 1.0000 on both the CICIoT network intrusion dataset and the Intel Berkeley sensor fault dataset. Recall = 1.000 across all 33 CICIoT attack categories confirms the model's ability to detect zero-day attacks without any labeled attack samples during training.
  2. 90% Bandwidth Reduction: The 10:1 compression ratio (320 → 32 values) enables deployment over severely bandwidth-constrained IoT links. The semantic vector provides a meaningful latent representation, as confirmed by ROC-AUC = 1.0000 — the semantic space perfectly separates normal from anomalous behavior.
  3. Lightweight and Efficient Model: With 159,141 parameters and 205 windows/second CPU throughput (~5ms latency per window), the GRU Autoencoder is well-suited for edge deployment. The ablation study confirms GRU requires 24% fewer parameters than LSTM while achieving 88.4% higher F1-score under equal training conditions.
  4. Production-Ready System: Three deployment modes were implemented: CSV streaming gateway with JSONL logging, FastAPI REST API with Swagger documentation, and MQTT gateway for native IoT protocol integration.
  5. Robust Evaluation: Comprehensive evaluation includes per-dataset confusion matrices, ROC-AUC, per-attack-type Precision/Recall/F1 breakdown across 33 categories, balanced sampling methodology for imbalanced datasets, and a three-model ablation study (GRU vs. LSTM vs. MLP).

7.2. Future Work

  1. Real Network Traffic Integration: Replace the research dataset with live network traffic captured via Scapy or tshark, enabling real-world deployment validation.
  2. Adaptive Threshold: Implement a dynamic threshold that updates based on a rolling window of recent reconstruction errors, addressing concept drift as network traffic patterns evolve over time.
  3. Ensemble Detection: Combine the GRU Autoencoder with Isolation Forest or One-Class SVM to improve detection of stealthy low-volume attacks that currently show lower precision (e.g., Recon, Web attacks).
  4. Model Quantization: Apply INT8 quantization to reduce model size by 4× and increase inference throughput, enabling deployment on microcontrollers and ultra-low-power edge devices.
  5. Hardware Deployment: Deploy the complete system on Raspberry Pi or NVIDIA Jetson Nano for real-world edge testing and latency benchmarking under production conditions.
  6. Alerting Integration: Add email, Telegram, or webhook notification callbacks to the MQTT gateway for real-time security operations center (SOC) alerting.
  7. Multi-Sensor Fusion: Extend the Intel Berkeley evaluation to fuse readings from multiple sensor nodes, capturing spatial anomaly patterns across the sensor network topology.
